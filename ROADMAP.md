# ppspecial / postpyc Roadmap

This roadmap is the coordination document for building `ppspecial` as a
complete postpyc implementation of `scipy.special`.

The goal is not only to create a Python package. The goal is to prove that
postpyc can be used to write one numerical library source base that
supports:

- interpreted execution under CPython,
- ahead-of-time compilation to a plain C-compatible shared library,
- a separate NumPy ufunc extension layer built on top of the same kernels,
- cooperation between multiple compiler implementations that target the
  postpyc standard.

`ppspecial` should stay clean postpyc. Compiler and specification gaps
discovered here should be reported and fixed in
the postpyc project.

## Working Rules

- Keep `ppspecial` source free of compiler-specific escape hatches.
- Use a locally checked-out postpyc repository for compiler verification
  during development.
- Keep that postpyc checkout updated regularly to the `main` branch.
- Before using a local compiler checkout for verification, confirm it is on
  `main`:

  ```bash
  git -C /path/to/postpyc status --short --branch
  git -C /path/to/postpyc rev-parse --abbrev-ref HEAD
  ```

- If the postpyc checkout is not on `main`, do not treat compiler results
  as canonical for ppspecial progress.
- Keep the plain C shared library target independent from the NumPy extension
  target. NumPy wrappers should delegate to the same compiled scalar/core
  kernels used by the C ABI artifact.
- Each target below should be small enough to land as a reviewable pull
  request with tests and a status update in this file.

## Status Legend

- `Done`: implemented and verified.
- `Active`: currently being worked on.
- `Ready`: scoped and ready for an agent/person to pick up.
- `Blocked`: waiting on postpyc compiler/spec work or an external decision.
- `Later`: intentionally deferred.

## Target 0: Baseline Package Health

Status: `Active`

Purpose: keep the existing package reliable while larger work proceeds.

Acceptance criteria:

- Interpreted tests pass:

  ```bash
  python -m pytest tests/
  ```

- Native build passes with a local postpyc checkout on `main`:

  ```bash
  PYTHONPATH=/path/to/postpyc python scripts/build_native.py
  ```

- `scripts/build_native.py` reports:

  - `_erf` builds,
  - `_bessel` builds,
  - `_gamma` builds,
  - `_stats` builds,
  - `_hyper` builds,
  - full `ppspecial` package shared library builds.

Current notes:

- All five current modules compile natively.
- The full package builds as a single shared library.
- README wording should remain precise: the package contains all lowered
  kernels, but the stable C export ABI is not final yet.

## Target 1: Native C ABI Artifact

Status: `Done` (verified against postpyc `16b7692`)

Purpose: produce an independent shared library callable from C, C++, Rust,
Go, Zig, Julia, R, ctypes, cffi, and other C-compatible interfaces.

This target is intentionally separate from NumPy ufunc extension generation.

Deliverables:

- A repeatable build command for a plain shared library, for example:

  ```bash
  python scripts/build_native.py
  ```

- A generated or handwritten C header describing the exported ABI.
  → `scripts/build_native.py` asks postpyc to emit `ppspecial.h`.
- A minimal `ctypes` smoke-test suite that loads the compiled package
  artifact and calls representative functions.
  → `tests/test_native_abi.py` loads the package shared library with
  `ctypes.CDLL` and calls representative unary, binary, and mixed int/float
  kernels through stable `pp_*` exports.
- Documentation that explains symbol names, aliases, and current ABI limits.
  → README documents `pp_*` C ABI exports plus the generated header and
  manifest sidecars.

Acceptance criteria:

- The shared library can be loaded with `ctypes.CDLL`.
- Representative functions from each family can be called through C ABI:

  - `erf`, `erfc`, `erfinv`,
  - `lgamma`, `gamma`, `digamma`,
  - `j0`, `j1`, `i0`, `k0`,
  - `ndtr`, `ndtri`, `expit`, `logit`.

- Tests distinguish source-level Python names from compiler-internal kernel
  symbols and call the stable `pp_*` C ABI exports. ✓

postpyc requests generated by this target (delivered upstream in
[`postpyc#12`](https://github.com/openteams-ai/postpyc/issues/12)):

- Stable package C ABI mode
  ([postpyc#12](https://github.com/openteams-ai/postpyc/issues/12)).
- Export manifest or compiler-visible equivalent of `__all__`
  ([postpyc#12](https://github.com/openteams-ai/postpyc/issues/12)).
- Native alias policy for names such as `gammaln = lgamma` and
  `sigmoid = expit`.
- C header generation for exported functions
  ([postpyc#12](https://github.com/openteams-ai/postpyc/issues/12)).
- Module-qualified native symbols, or another stable collision-free ABI
  ([postpyc#12](https://github.com/openteams-ai/postpyc/issues/12)).

## Target 2: NumPy Ufunc Extension Layer

Status: `Done` (verified against postpyc `ecb13fd`)

Purpose: build a Python extension module exposing NumPy ufuncs while keeping
the core kernels reusable outside Python.

Deliverables:

- Separate extension build target, distinct from the plain C shared library.
  → `scripts/build_ext.py` / `pixi run build-ext` builds `ppspecial_native`.
- NumPy ufunc registration for scalar `@vectorize` kernels.
  → all 28 public non-alias ufuncs register (element-wise and gufunc forms).
- Tests that compare compiled ufunc calls with interpreted postpyc calls.
  → `tests/test_native_ext.py`: per-function grids at rtol 1e-13, plus
  broadcasting, `out=`, mixed int/float dtypes, and docstring checks.
- Documentation explaining the relationship between:

  - postpyc source,
  - native C ABI kernels,
  - NumPy ufunc wrappers,
  - Python package imports.

  → README "Compiled extension module (NumPy ufuncs)" section.

Acceptance criteria:

- `import ppspecial_native` or equivalent exposes NumPy-callable ufuncs. ✓
- Array broadcasting behavior matches interpreted mode for current functions. ✓
- The extension layer delegates to the same compiled kernels as the plain
  shared library target. ✓ (same translation-unit objects, linked with a
  generated PyInit shim)

postpyc requests generated by this target (all delivered upstream in
openteams-ai/postpyc#10):

- Extension-module backend target. → `build_file(..., ext_module=True)`
- NumPy ufunc registration boilerplate generation. → generated PyInit shim
- Clear separation between kernel ABI and Python extension ABI. → only the
  shim needs Python/NumPy headers; translation units stay pure C
- Build API that can emit both native library and extension module artifacts
  from the same postpyc package. → same `build_file`, flag-selected

## Target 3: Accuracy and Compatibility Harness

Status: `Ready`

Purpose: make numerical accuracy a first-class, published part of the
project rather than a collection of ad hoc tests.

Deliverables:

- Per-function accuracy table with:

  - domain covered,
  - reference source,
  - absolute tolerance,
  - relative tolerance,
  - known edge cases,
  - current max observed error.

- Optional SciPy-backed test mode when SciPy is installed.
- Deterministic reference-value tests that do not require SciPy.
- Compiled-vs-interpreted consistency tests.

Acceptance criteria:

- Every implemented function has documented accuracy expectations.
- Tests cover special values where meaningful:

  - `0`,
  - `-0`,
  - infinities,
  - NaN propagation,
  - poles,
  - branch cuts or domain boundaries.

- Accuracy regressions are easy to attribute to either algorithm changes or
  compiler/backend changes.

postpyc requests generated by this target:

- Better floating-point semantics documentation.
- Explicit behavior for NaN, infinities, signed zero, and libm dependence.
- Optional fast-math controls, with default strict behavior.

## Target 8: Hypergeometric Functions

Status: `Active`

Candidate functions:

- `hyp0f1`
- `hyp1f1`
- `hyp2f1`
- `hyperu`

Suggested first slice:

- Implement `hyp0f1` for real `f64`.
- Add reference tests over a conservative real domain.
- Use the work to identify loop, convergence, and error-handling needs in
  postpyc.

Current first-slice notes:

- `hyp0f1(b, x)` is implemented by the defining power series for real `f64`.
- Deterministic tests cover the SciPy-documented `hyp0f1(1, 1)` value plus
  cosine/sine and hyperbolic identities for `b = 1/2` and `b = 3/2`.
- The scalar kernel intentionally uses a bounded convergence loop and current
  ppspecial sentinel conventions rather than compiler-specific escape hatches.
- SciPy/XSF's real implementation evaluates `hyp0f1` through Bessel/gamma
  identities plus an asymptotic fallback. This first slice uses the defining
  series because `ppspecial` does not yet provide arbitrary-order `jv`/`iv`,
  `gammasgn`, `sinpi`, or standardized NaN/overflow semantics. Those
  prerequisites should land before replacing the series kernel with an
  XSF-style piecewise implementation.

postpyc requests generated or confirmed by this target:

- More robust loop lowering and optimization.
- Better handling of convergence criteria, including a standard way for a
  scalar kernel to report non-convergence instead of returning the final
  partial sum from a hard iteration cap.
- Standardized domain/error signaling for singular bottom parameters and
  overflow, beyond the current package-local `±1.0e308` sentinel convention.
- Possible support for local helper functions with inlining hints as the
  hypergeometric family grows beyond the single direct-series kernel.

## Target 11: Packaging, CI, and Release Flow

Status: `Active`

Deliverables:

- CI job for interpreted tests.
- CI job for native build using postpyc from `main`.
- Optional CI job for SciPy comparison tests.
- Published build status in README.
- Source-only PyPI releases: pure `py3-none-any` wheels and sdists, with no
  binary wheels and no install-time/import-time compilation.
- Package-manager layout for compiled artifacts:

  - `libppspecial`: shared library, `ppspecial.h`, and POST ABI manifest.
  - `ppspecial`: Python source package plus optional NumPy ufunc extension.

- `pixi run build-prefix` / package recipes use:

  ```bash
  postpyc build ppspecial/__init__.py --prefix "$PREFIX" --module-name ppspecial
  ```

- Versioned release notes that distinguish:

  - source-level Python API,
  - plain native C ABI,
  - NumPy extension ABI.

Acceptance criteria:

- A contributor can run one command for interpreted tests.
- A contributor can run one command for native compilation.
- A package recipe can run one command to produce the `libppspecial` prefix
  layout.
- CI makes it clear whether a failure is in ppspecial, postpyc compiler,
  packaging, or optional SciPy comparison.

postpyc requests generated by this target:

- Stable CLI for compiler invocation. (done: postpyc distribution CLI)
- Stable build API return metadata: generated C files, object files, linked
  artifacts, exported symbols, diagnostics. (partly done via ABI manifest)
- Machine-readable compiler diagnostics.
